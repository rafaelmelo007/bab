# PRD — `bab`

**A unified CLI gateway to Claude, Codex, Gemini, and Ollama.**

**Status:** Draft for review
**Owner:** TBD
**Last updated:** 2026-05-23

---

## 1. Summary

`bab` is a single command-line tool that routes prompts to one of four LLM providers — **Claude**, **Codex** (OpenAI), **Gemini**, or **Ollama** — by invoking each provider's official CLI as a subprocess. The user sets the active provider once with a `/provider` command and continues chatting normally; every subsequent prompt goes to that provider until the user switches.

The product reuses the user's existing subscriptions (Claude Max, ChatGPT Plus, Gemini Advanced) and local Ollama install rather than requiring API keys.

### Why "bab"

From the Akkadian *bāb* — "gate." The root of Babylon (*Bāb-ilim*, "gate of the god"), and a living word in Arabic, Persian, Hebrew, and Turkish meaning gate, doorway, or chapter. The metaphor fits the product precisely: one gateway, many destinations beyond it.

---

## 2. Goals

- **One gateway, four backends.** No mental overhead about which CLI to launch.
- **Subscription-first.** Leverage existing logins, not API keys.
- **Sticky context.** Pick a provider, keep using it without re-typing flags.
- **Minimal surface area.** A handful of slash commands, nothing more.

## 3. Non-goals

- Building a chat UI, TUI, or web frontend.
- Reimplementing provider features (file editing, tool use, agentic loops) — those stay inside each provider's CLI.
- Cross-provider conversation memory or context sharing.
- Token counting, cost tracking, or model benchmarking.
- Plugin system or extensibility hooks (v1).

---

## 4. User experience

### 4.1 Launch

```
$ bab
bab> _
```

A persistent REPL. No provider selected yet — the user must pick one.

### 4.2 Setting the provider

```
bab> /provider claude
✓ provider set to claude
bab (claude)> _
```

The prompt prefix updates to show the active provider. Every plain message from here goes to Claude until the user switches.

### 4.3 Sending a prompt

```
bab (claude)> explain monads in one paragraph
[claude] Monads are a way to chain computations that carry context...
bab (claude)> _
```

The user types, hits enter, the response streams back inline.

### 4.4 Switching providers

```
bab (claude)> /provider gemini
✓ provider set to gemini
bab (gemini)> what about in haskell specifically?
[gemini] In Haskell, monads are...
```

Note: conversation history does **not** carry across providers. Each provider keeps its own session.

### 4.5 Slash commands (the whole list)

| Command | Behavior |
|---|---|
| `/provider <name>` | Set active provider. Valid: `claude`, `codex`, `gemini`, `ollama` |
| `/provider` | Show current provider |
| `/providers` | List available providers and their status (logged in / not configured) |
| `/new` | Start a fresh session with the current provider (drops stored session ID) |
| `/sessions` | List recent session IDs for the current provider |
| `/resume <id>` | Resume a specific past session with the current provider |
| `/model <name>` | Override the model for the current provider (e.g. `/model opus`, `/model llama3`) |
| `/clear` | Clear the screen |
| `/help` | Show commands |
| `/exit` or Ctrl-D | Quit |

That's it. No config files to learn, no flags to memorize.

### 4.6 One-shot mode

For scripts and pipes:

```bash
$ bab -p claude "summarize this" < notes.txt
$ echo "rewrite in french" | bab -p gemini
```

Same behavior, single round-trip, exits when done.

## 5. How it works (technical sketch)

### 5.1 Provider invocation

Each provider is driven by spawning its official CLI as a subprocess. Three transport modes, in order of preference:

1. **ACP (Agent Client Protocol)** — JSON-RPC 2.0 over stdio. Spawn once, keep alive for the session. Used where supported.
2. **Non-interactive exec flag** — `claude -p`, `codex exec`, `gemini -p`. New process per turn. Session continuity via `--resume <id>` where available.
3. **HTTP** — Ollama only. Talks to `http://localhost:11434` directly.

`bab` detects which mode each installed CLI supports on first run and caches the result.

### 5.2 Session state

- `bab` itself stores **no conversation content**.
- Session IDs returned by each provider CLI are tracked per-provider so `/new` can issue a fresh one and normal turns can resume the existing one.
- Switching providers does not merge histories. Each provider sees only its own thread.

### 5.3 Auth

- `bab` never handles credentials. If a provider CLI is not logged in, `bab` surfaces its error and tells the user to run the provider's own login command (`claude login`, etc.).
- No API keys are stored or read by `bab`.

### 5.4 Discovery

On first launch:
- Probe `$PATH` for `claude`, `codex`, `gemini`, `ollama`.
- For each found, run a lightweight version check.
- Report missing providers in `/providers` output. Don't block startup — users may only want one.

## 5.5 Memory & session model

`bab` is deliberately thin on memory. Four kinds of memory are in play, with distinct policies:

### 5.5.1 Within-session conversation history

**Owned by the provider CLI. `bab` stores none of it.**

When the user sends a prompt, the provider CLI maintains the turn-by-turn history. `bab` is a conduit — it forwards prompts and streams responses, but never inspects, caches, or replays conversation content.

### 5.5.2 Session continuity across `bab` restarts

**Tracked by `bab` as session IDs only. Content stays with the provider.**

Each provider CLI assigns a session ID per conversation (e.g. Claude Code in `~/.claude/projects/`, similar paths for Codex and Gemini). `bab` stores the **last session ID per provider** in `~/.config/bab/state.toml`:

```toml
[sessions]
claude = "abc-123-def"
codex  = "xyz-789-uvw"
gemini = "ghi-456-jkl"
ollama = null  # ollama is stateless via HTTP
```

On launch, when the user selects a provider, `bab` resumes that provider's last session by default. The user can override:

| Command | Behavior |
|---|---|
| `/new` | Discard the stored ID for the current provider and start fresh |
| `/sessions` | List recent session IDs from the provider CLI's storage |
| `/resume <id>` | Jump into a specific past session |

What `bab` stores: ID strings and timestamps. **Never conversation content.**

### 5.5.3 Cross-provider memory

**Explicit non-goal.** Conversations do not bleed between providers. Switching from Claude to Gemini gives Gemini a clean slate. Rationale:

- Each provider has different context windows, pricing, and behavior; silent context injection would create surprise costs and surprise truncation.
- Each provider's session storage is its own; merging would require `bab` to handle conversation content, which it deliberately avoids.
- The "magic" of automatic context transfer is hard to debug when it misfires.

Users who need cross-provider context can copy-paste manually. A future `/pipe` command is listed in v2 candidates.

### 5.5.4 User preferences & persistent context

**Use the provider's native context files. `bab` does not add its own.**

Each provider already supports a persistent instruction file:

- Claude Code: `CLAUDE.md`
- Gemini CLI: `GEMINI.md`
- Codex CLI: `AGENTS.md` or `codex.md`
- Ollama: model-level system prompts

Adding a `bab.md` layer would create a fourth place to put instructions and risk contradicting the provider-specific files. The minimal-surface-area principle says no.

What `bab` *does* store in `~/.config/bab/state.toml`:

```toml
[preferences]
default_provider  = "claude"  # last used, if 6.5.3 option chosen
default_models    = { }       # per-provider model overrides set via /model
```

Nothing else. No prompts, no history, no profile, no API keys.

## 6. Constraints & open questions

### Constraints

- Each provider's CLI must be installed by the user. `bab` does not install them.
- Cold-start latency in exec mode is ~300ms–1.5s per turn. Acceptable for interactive use; documented for scripting.
- Output is streamed verbatim from the provider CLI. Markdown rendering, syntax highlighting, etc. are whatever each CLI produces.

### Open questions

1. **Terms of service** — confirm each provider permits programmatic invocation of their CLI under subscription. Document any restrictions in README.
2. **PTY fallback** — do we ship a vt100 screen-scraper for CLIs that lack ACP and lack a clean `-p` mode? Adds complexity; defer to v2 unless needed at launch.
3. **Default provider** — should `bab` remember the last-used provider across sessions, or always start clean? Lean toward remembering, stored in `~/.config/bab/state.toml`.
4. **Model defaults** — per-provider default model, or always inherit the provider CLI's default? Lean toward inheriting; expose `/model` for overrides.
5. **Error surface** — when a subprocess crashes mid-stream, do we retry, fail loudly, or fall back? Lean toward fail-loud with a clear message.
6. **Implementation language** — Rust (fast cold start, single static binary), Go (similar profile, simpler ecosystem), or Python (faster iteration, slower startup). Lean Rust given gate4agent prior art and the cold-start sensitivity.

## 6.5 Future architecture: optional daemon mode

A daemon mode is **out of scope for v1** but is the most likely v2 direction. This section exists to ensure the v1 design does not foreclose it.

### Motivation

In v1, every `bab` invocation spawns the provider CLI fresh, paying 300ms–1.5s of cold-start cost per turn. A long-lived `bab` daemon would keep ACP-mode provider CLIs warm across invocations, dropping first-token latency to near-zero and enabling cross-shell session persistence.

### Shape (sketch only)

```
┌─────────────┐         ┌──────────────────────────┐
│  bab CLI    │ ──IPC──▶│  bab daemon (background) │
│  (client)   │         │  ├── claude --acp (warm) │
└─────────────┘         │  ├── codex --acp  (warm) │
                        │  ├── gemini --acp (warm) │
                        │  └── ollama HTTP client  │
                        └──────────────────────────┘
```

- IPC over a Unix domain socket (or named pipe on Windows), JSON-RPC framing.
- Daemon managed via `bab daemon start|stop|status|logs|install|uninstall`.
- Optional auto-start via launchd / systemd / Windows service.

### Non-negotiable principles for the daemon path

1. **The daemon adds speed, never features.** Anything `bab` can do with the daemon, it must do without. The daemon is a performance layer, not a capability gate.
2. **Graceful degradation.** If the daemon is not running or its socket is unreachable, `bab` falls back to direct subprocess spawning (v1 behavior) silently — at most a one-line warning.
3. **Auto-detect must be cheap.** The "is there a daemon?" check on each CLI invocation must complete in under 10ms (socket existence + one-byte ping), so the no-daemon path is never penalized.
4. **Stateless CLI, stateful daemon.** The CLI binary stores nothing. All session state lives in the daemon or the provider CLIs it manages. CLI upgrades must not require daemon restart.
5. **Daemon is always optional.** Some users will never want a background process. They must get a great experience anyway.

### v1 design implications

To keep the daemon path open without building it now:

- **Abstract the provider transport.** The code that spawns and communicates with a provider CLI should sit behind an interface (e.g. `ProviderTransport`) with one v1 implementation (direct subprocess) and a placeholder for a future second implementation (daemon socket client). Swapping should be a single-file change.
- **Keep session IDs externally addressable.** Don't bury session identifiers inside in-memory state that only the current process can see. They should be retrievable strings, suitable for handoff to a daemon later.
- **Don't store transient state in the CLI process** beyond what's needed for the current REPL session. Anything that would need to survive a CLI restart belongs in a clearly separated layer that a daemon could own in v2.

No daemon code ships in v1. These are constraints on v1 to avoid painting into a corner.

## 7. Success criteria

- A user with all four CLIs installed can switch providers and send 10 prompts in under 30 seconds with no documentation lookup.
- Cold install → first successful prompt in under 2 minutes assuming provider CLIs are already logged in.
- Zero credential handling code in `bab` (verified by audit).
- Single binary, under 10MB, no runtime dependency beyond the provider CLIs themselves.

---

## 8. Out of scope for v1 (candidate v2 features)

- **Daemon mode** with warm provider CLIs, cross-shell session persistence, and IPC over a local socket (see §6.5).
- Parallel fan-out (`/all explain X` → query all four, show side-by-side).
- Cross-provider piping (`/pipe claude → gemini`).
- Saved prompts / aliases.
- Conversation export.
- Config file for power users.
