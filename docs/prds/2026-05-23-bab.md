## ¬ß1 Product Vision

`bab` is a single command-line tool that routes prompts to one of four LLM providers ‚Äî **Claude**, **Codex** (OpenAI), **Gemini**, or **Ollama** ‚Äî by invoking each provider's official CLI as a subprocess. The user sets the active provider once with a `/provider` command and continues chatting normally; every subsequent prompt goes to that provider until the user switches.

The product reuses the user's existing subscriptions (Claude Max, ChatGPT Plus, Gemini Advanced) and local Ollama install rather than requiring API keys. The bet: developers who already pay for two or three of these subscriptions waste real time switching terminals and re-typing context; one persistent gate erases that friction without trying to be a meta-provider or build new capability.

### Why "bab"

From the Akkadian *bƒÅb* ‚Äî "gate." The root of Babylon (*BƒÅb-ilim*, "gate of the god"), and a living word in Arabic, Persian, Hebrew, and Turkish meaning gate, doorway, or chapter. The metaphor fits the product precisely: one gateway, many destinations beyond it.

## ¬ß4 User Experience

### ¬ß4.1 Launch

```
$ bab
bab> _
```

A persistent REPL. No provider selected yet ‚Äî the user must pick one.

### ¬ß4.2 Setting the provider

```
bab> /provider claude
‚úì provider set to claude
bab (claude)> _
```

The prompt prefix updates to show the active provider. Every plain message from here goes to Claude until the user switches.

### ¬ß4.3 Sending a prompt

```
bab (claude)> explain monads in one paragraph
[claude] Monads are a way to chain computations that carry context...
bab (claude)> _
```

The user types, hits enter, the response streams back inline.

### ¬ß4.4 Switching providers

```
bab (claude)> /provider gemini
‚úì provider set to gemini
bab (gemini)> what about in haskell specifically?
[gemini] In Haskell, monads are...
```

Note: conversation history does **not** carry across providers. Each provider keeps its own session.

### ¬ß4.5 Slash commands (the whole list)

| Command | Behavior |
|---|---|
| `/provider <name>` | Set active provider. Valid: `claude`, `codex`, `gemini`, `ollama` |
| `/provider` | Show current provider |
| `/providers` | List available providers and their status (logged in / not configured) |
| `/new` | Start a fresh session with the current provider (drops stored session ID) |
| `/sessions` | List recent session IDs for the current provider |
| `/resume <id>` | Resume a specific past session with the current provider |
| `/model <name>` | Override the model for the current provider (e.g. `/model opus`, `/model llama3`) |
| `/clear` | Clear the screen |
| `/help` | Show commands |
| `/exit` or Ctrl-D | Quit |

10 commands; under the G-04 cap of 12. No config files to learn, no flags to memorize.

### ¬ß4.6 One-shot mode

For scripts and pipes:

```bash
$ bab -p claude "summarize this" < notes.txt
$ echo "rewrite in french" | bab -p gemini
```

Same behavior, single round-trip, exits when done.

### ¬ß4.7 UX baseline

These are the conventions every command in ¬ß4.5 must follow. They exist so frontend-lead and ux-specialist can score against something concrete rather than infer from happy-path examples.

- **TTY detection & color.** Respect `NO_COLOR` env var (disables all ANSI). Auto-disable color when stdout is not a TTY (e.g. one-shot mode piped into a file). Honor `FORCE_COLOR` to override. Flag `--no-color` provided.
- **UTF-8 with ASCII fallback.** Glyphs (`‚úì`, `‚úó`, `‚öÝ`, `‚Ýã‚Ýô‚Ýπ‚Ý∏` spinner) downgrade to ASCII (`[ok]`, `[err]`, `[!]`, `|/-\`) when `LANG`/`LC_ALL` does not include UTF-8 or the legacy Windows console is detected.
- **Streaming behavior.** Provider output is streamed to stdout as it arrives. No spinner during streaming. A spinner shows only between prompt-submit and first-byte-received.
- **Ctrl-C semantics.** First Ctrl-C cancels the current turn ‚Äî SIGINT to subprocess (or `CTRL_BREAK_EVENT` on Windows), 500 ms grace, SIGTERM/`TerminateProcess`, 500 ms, SIGKILL. Returns to REPL. Second Ctrl-C within 1 s exits bab.
- **Unknown command.** `/foo` ‚Üí `‚úó unknown command: /foo. Try /help.` If Levenshtein distance ‚â§ 2 from a known command, append `Did you mean /<closest>?`. Tie-break: alphabetical (earliest in `/help` list wins).
- **History.** Up/Down arrow keys cycle through previous prompts in the current REPL session (in-memory; not persisted across restarts in v1).
- **Line editing.** Standard readline-style bindings via the `reedline` crate: Ctrl-A/E (line start/end), Ctrl-W (kill word), Ctrl-U (kill line), Ctrl-K (kill to EOL), Alt-‚Üê/‚Üí (word jump). Cursor movement on resize: SIGWINCH triggers full prompt re-render; long lines wrap rather than truncate.
- **Streaming on resize.** SIGWINCH during active streaming does not interrupt the subprocess. Output continues to flow; the next prompt is re-rendered at the new width.
- **Tab completion.** Out of scope for v1; tracked as v2 candidate in ¬ß15.
- **Spinner cadence.** 80 ms per frame (12.5 fps), minimum 200 ms display time so sub-200ms first-byte responses don't flicker the spinner.
- **First-run screen.** When `~/.config/bab/state.toml` does not exist:
  ```
  Welcome to bab.
  Detected providers: claude ‚úì  codex ‚úó (not installed)  gemini ‚úì  ollama ‚úì
  Set one with /provider <name>. /help for the full command list.
  ```
  When **zero** providers are detected:
  ```
  Welcome to bab.
  No providers found on $PATH. Install at least one:
    claude:  https://docs.anthropic.com/claude-code
    codex:   https://github.com/openai/codex
    gemini:  https://github.com/google-gemini/gemini-cli
    ollama:  https://ollama.com/download
  Re-run bab once installed.
  ```
- **No-provider prompt.** If the user sends a plain message before running `/provider`, bab prints `‚úó no provider selected. Run: /provider <name>` and returns to the prompt without invoking any subprocess.

Three columns, fixed-width: `ID` (12 chars truncated with `‚Ä¶` if longer), `LAST_USED` (RFC3339 UTC, 20 chars), `PREVIEW` (40 chars from the first user message of the session, truncated with `‚Ä¶`). ## ¬ß5 How it works (technical sketch)

### ¬ß5.1 Provider invocation

Each provider is driven by spawning its official CLI as a subprocess. Three transport modes, in order of preference per provider:

1. **ACP (Agent Client Protocol)** ‚Äî JSON-RPC 2.0 over stdio. Spawn once, keep alive for the session. Used where supported.
2. **Non-interactive exec flag** ‚Äî `claude -p`, `codex exec`, `gemini -p`. New process per turn. Session continuity via `--resume <id>` where available.
3. **HTTP** ‚Äî Ollama only. Talks to `http://localhost:11434` directly.

#### ¬ß5.1.1 Per-provider transport capability matrix

| Provider | ACP support | Exec flag | HTTP | v1 transport |
|----------|-------------|-----------|------|--------------|
| claude   | TBD (PRE-02) | `claude -p` | ‚Äî | ACP if PRE-02 passes, else exec |
| codex    | TBD (PRE-02) | `codex exec` | ‚Äî | exec (v1); ACP deferred to v2 |
| gemini   | TBD (PRE-02) | `gemini -p` | ‚Äî | exec (v1); ACP deferred to v2 |
| ollama   | ‚Äî | ‚Äî | `localhost:11434` | HTTP |

The matrix is the source of truth ‚Äî `bab` does not "auto-detect mode" beyond what this table prescribes. Capability detection (¬ß5.4) confirms the chosen transport works; failure surfaces `E-ACP-PROTOCOL` and falls back to exec one turn at a time.

#### ¬ß5.1.2 Subprocess lifecycle contract

- **Spawn.** `Command::new("<provider>").args(&[...]).stdin(piped()).stdout(piped()).stderr(piped()).spawn()`. **Never via a shell.** Argv array only ‚Äî no `bash -c`, no `cmd /c`. Audited by a CI grep in ¬ß7.2.
- **Stdio framing.** stdout streamed line-by-line, UTF-8. stderr buffered, printed at process exit. ACP mode: length-prefixed JSON-RPC frames per ACP spec.
- **Timeout.** Exec mode: 60 s per turn default (`BAB_TURN_TIMEOUT` override). ACP mode: no aggregate cap, but 30 s no-byte-received kills the subprocess.
- **Cancellation.** ¬ß4.7 Ctrl-C semantics.
- **Cleanup on REPL exit.** All child processes (ACP warm ones included) killed. PID marker written to `~/.cache/bab/pids/<provider>.pid` with the process start-time (via `/proc/<pid>/stat` field 22 on Linux, `kinfo_proc` on macOS, `GetProcessTimes` on Windows). On startup, bab reaps stale PIDs **only if** the start-time still matches ‚Äî defeats PID-reuse attacks where an unrelated process now holds the same PID.
- **ACP keepalive.** ACP mode sends a `ping` JSON-RPC method every 10 s when no other traffic is in flight. The "30 s no-byte-received" kill rule applies only when no `ping` response has been received either; legitimate long-running provider thinks do not trigger it.
- **Partial streams.** Mid-stream crash ‚Üí bab prints `--- provider 'X' exited (N) mid-stream ---` and returns to prompt with `E-CLI-CRASH`.

### ¬ß5.2 Session state

- `bab` itself stores **no conversation content**.
- Session IDs returned by each provider CLI are tracked per-provider so `/new` can issue a fresh one and normal turns can resume the existing one.
- Switching providers does not merge histories. Each provider sees only its own thread.

#### ¬ß5.2.1 `state.toml` schema, versioning, atomic writes

- Every state.toml starts with `schema_version = 1`.
- bab refuses to read schemas it does not recognize (`E-STATE-SCHEMA`). Migrations live in `src/state/migrate.rs`; running bab transparently writes back the new schema after a successful migration. Unknown keys are preserved on rewrite (forward-compat).
- **Atomic write.** Write to `state.toml.tmp`, fsync, rename over `state.toml`. Lockfile `state.toml.lock` (OS-level flock / `LockFileEx` on Windows) prevents concurrent writers; readers do not lock. Retry on contention: 3 attempts, 50 ms backoff, then `E-STATE-LOCKED`.
- **Read-modify-write merge.** Within the write lock, bab re-reads the on-disk state and merges its in-memory changes (per-key, not whole-file) before writing. Unknown keys preserved. This prevents a one-shot from silently dropping a concurrent REPL's session-ID write.
- **Migration safety.** Before applying a schema migration, bab copies `state.toml` to `state.toml.v<old>.bak`. Migration runs in a tmp file; rename only after success. Partial-failure ‚Üí bak is restored on next bab launch; user sees a one-line warning naming both files.
- **File mode.** `0600` on Unix; checked on every load (`E-PERMS` if more permissive). On Windows: NTFS ACL grants Full Control to the current user only, removes BUILTIN\Users.
- **Concurrent invocations.** A REPL plus a parallel `bab -p ... "..."` one-shot is supported ‚Äî both read on entry; per-key merge (above) preserves both writers' session IDs. mtime/etag mismatch between read and write triggers a one-line warning.
- **Model-id validation.** `default_models` values are stored as opaque strings ‚Äî bab does not validate model IDs (provider CLIs do). Unknown / typo'd model IDs surface as the provider's own error on first use of that provider.

### ¬ß5.3 Auth

- `bab` never handles credentials. If a provider CLI is not logged in, `bab` surfaces its error and tells the user to run the provider's own login command (`claude login`, etc.).
- No API keys are stored or read by `bab`.
- Subprocess stderr is scanned for known credential shapes (Bearer tokens, OAuth refresh URLs, basic-auth headers) and redacted before printing ‚Äî see ¬ß7.2 log redaction.

### ¬ß5.4 Discovery

On first launch:
- Probe `$PATH` for `claude`, `codex`, `gemini`, `ollama` via a `which`-equivalent. Cache the **absolute** resolved path in state.toml.
- For each found, run a lightweight `<provider> --version` and record the version string.
- Discovery probe timeout: 500 ms per provider. Total first-run discovery budget: 1500 ms wall-clock.
- Report missing providers in `/providers` output. Don't block startup ‚Äî users may only want one.

**PATH-poisoning defense.** Every subsequent invocation re-verifies the cached absolute path still exists. If missing, re-probe. If the resolved binary changes (different absolute path), invalidate the cached capability detection and re-run ¬ß5.1.1 transport check. **Integrity check.** bab additionally records the SHA-256 of each provider binary at first detection in `state.toml`. Hash mismatch ‚Üí re-run capability detection and surface a one-line `‚öÝ provider 'X' binary changed since last detection` warning. Hash check is informational, not a security boundary: a malicious replacement at the same path that updates the hash record concurrently is not defended against ‚Äî that requires OS-level binary signing.

**Version-skew handling.** If `<provider> --version` differs from the cached version, re-run capability detection (a provider may have gained/lost ACP support).

## ¬ß5.5 Memory & session model

`bab` is deliberately thin on memory. Four kinds of memory are in play, with distinct policies:

### ¬ß5.5.1 Within-session conversation history

**Owned by the provider CLI. `bab` stores none of it.**

When the user sends a prompt, the provider CLI maintains the turn-by-turn history. `bab` is a conduit ‚Äî it forwards prompts and streams responses, but never inspects, caches, or replays conversation content.

### ¬ß5.5.2 Session continuity across `bab` restarts

**Tracked by `bab` as session IDs only. Content stays with the provider.**

Each provider CLI assigns a session ID per conversation (e.g. Claude Code in `~/.claude/projects/`, similar paths for Codex and Gemini). `bab` stores the **last session ID per provider** in `~/.config/bab/state.toml`:

```toml
schema_version = 1

[sessions]
claude = "abc-123-def"
codex  = "xyz-789-uvw"
gemini = "ghi-456-jkl"
ollama = ""  # ollama is stateless via HTTP; empty string, not null
```

On launch, when the user selects a provider, `bab` resumes that provider's last session by default. The user can override:

| Command | Behavior |
|---|---|
| `/new` | Discard the stored ID for the current provider and start fresh |
| `/sessions` | List recent session IDs from the provider CLI's storage |
| `/resume <id>` | Jump into a specific past session |

What `bab` stores: ID strings and timestamps. **Never conversation content.**

**Coupling to provider storage.** `/sessions` and `/resume` read each provider CLI's own session store directly. If a provider changes its storage layout, `/sessions` and `/resume` break for that provider until bab is updated. Tracked via PRE-03 and the ¬ß14.1 minimum-version pin.

### ¬ß5.5.3 Cross-provider memory

**Explicit non-goal.** Conversations do not bleed between providers. Switching from Claude to Gemini gives Gemini a clean slate. Rationale:

- Each provider has different context windows, pricing, and behavior; silent context injection would create surprise costs and surprise truncation.
- Each provider's session storage is its own; merging would require `bab` to handle conversation content, which it deliberately avoids.
- The "magic" of automatic context transfer is hard to debug when it misfires.

Users who need cross-provider context can copy-paste manually. A future `/pipe` command is listed in ¬ß15 v2 candidates.

### ¬ß5.5.4 User preferences & persistent context

**Use the provider's native context files. `bab` does not add its own.**

Each provider already supports a persistent instruction file:

- Claude Code: `CLAUDE.md`
- Gemini CLI: `GEMINI.md`
- Codex CLI: `AGENTS.md` or `codex.md`
- Ollama: model-level system prompts

Adding a `bab.md` layer would create a fourth place to put instructions and risk contradicting the provider-specific files. The minimal-surface-area principle says no.

What `bab` *does* store in `~/.config/bab/state.toml`:

```toml
schema_version = 1

[preferences]
default_provider = "claude"      # per Q-03 resolution: remember last-used
default_models   = { claude = "claude-opus-4-7", gemini = "gemini-2.0-pro" }
# default_models is a TOML inline table: keys are provider names, values are model-id strings.
# Unknown keys are preserved on write (forward-compat). Empty table {} = inherit provider's default.

[providers.claude]
absolute_path  = "/usr/local/bin/claude"
version        = "1.4.2"
transport      = "acp"              # one of: "acp", "exec", "http"
detected_at    = "2026-05-23T14:02:11Z"

# ... one [providers.X] block per detected provider ...
```

Nothing else. No prompts, no history, no profile, no API keys.

## ¬ß9 Future architecture: optional daemon mode

A daemon mode is **out of scope for v1** but is the most likely v2 direction. This section exists to ensure the v1 design does not foreclose it.

### ¬ß9.1 Motivation

In v1, every `bab` invocation spawns the provider CLI fresh, paying 300 ms‚Äì1.5 s of cold-start cost per turn. A long-lived `bab` daemon would keep ACP-mode provider CLIs warm across invocations, dropping first-token latency to near-zero and enabling cross-shell session persistence.

### ¬ß9.2 Shape (sketch only)

```
‚îå‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îê         ‚îå‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îê
‚îÇ  bab CLI    ‚îÇ ‚îÄ‚îÄIPC‚îÄ‚îÄ‚ñ∂‚îÇ  bab daemon (background) ‚îÇ
‚îÇ  (client)   ‚îÇ         ‚îÇ  ‚îú‚îÄ‚îÄ claude --acp (warm) ‚îÇ
‚îî‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îò         ‚îÇ  ‚îú‚îÄ‚îÄ codex --acp  (warm) ‚îÇ
                        ‚îÇ  ‚îú‚îÄ‚îÄ gemini --acp (warm) ‚îÇ
                        ‚îÇ  ‚îî‚îÄ‚îÄ ollama HTTP client  ‚îÇ
                        ‚îî‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îò
```

- IPC over a Unix domain socket (or named pipe on Windows), JSON-RPC framing.
- Daemon managed via `bab daemon start|stop|status|logs|install|uninstall`.
- Optional auto-start via launchd / systemd / Windows service.

### ¬ß9.3 Non-negotiable principles for the daemon path

1. **The daemon adds speed, never features.** Anything `bab` can do with the daemon, it must do without. The daemon is a performance layer, not a capability gate.
2. **Graceful degradation.** If the daemon is not running or its socket is unreachable, `bab` falls back to direct subprocess spawning (v1 behavior) silently ‚Äî at most a one-line warning.
3. **Auto-detect must be cheap.** The "is there a daemon?" check on each CLI invocation must complete in under 10 ms (socket existence + one-byte ping), measured on the ¬ß7.1 benchmark hardware.
4. **Stateless CLI, stateful daemon.** The CLI binary stores nothing beyond its REPL session. All cross-shell session state lives in the daemon or the provider CLIs it manages. CLI upgrades must not require daemon restart.
5. **Daemon is always optional.** Some users will never want a background process. They must get a great experience anyway.

### ¬ß9.4 v1 design implications

To keep the daemon path open without building it now:

- **Abstract the provider transport.** F-02 sits behind a `ProviderTransport` interface with one v1 implementation (direct subprocess) and a placeholder for a future second implementation (daemon socket client). Swapping should be a single-file change.
- **Keep session IDs externally addressable.** Don't bury session identifiers inside in-memory state that only the current process can see. They are retrievable strings, suitable for handoff to a daemon later (already required by ¬ß5.2 atomic-write contract).
- **Don't store transient state in the CLI process** beyond what's needed for the current REPL session. Anything that would need to survive a CLI restart belongs in state.toml or provider storage, both of which a daemon could own in v2.

No daemon code ships in v1. These are constraints on v1 to avoid painting into a corner.

### ¬ß10.2 Telemetry (opt-in, off by default)

- Disabled until the user runs `bab telemetry enable`. Enabling prints a verbatim list of what is and isn't sent.

- **Local-only mode:** `bab telemetry local` writes events to `~/.cache/bab/telemetry.jsonl` for self-inspection. Never transmitted.
- **Disable:** `bab telemetry disable` purges any cached events and stops collection.
- **Kill switch:** `BAB_NO_TELEMETRY=1` overrides user config (for CI, sandboxes, etc.).
